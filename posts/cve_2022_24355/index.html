About TP-Link TL-WR940N and the vulnerability

The TP-Link TL-WR940N is a widely used wireless router known for its affordability and basic home networking features. In this article, i will reproduce CVE-2022-24355, a critical vulnerability on TL-WR940N firmware version 200316 and lower. CVE-2022-24355 allows attackers to execute malicious code on affected installations of TP-Link TL-WR940N routers, with no more authentication. This vulnerability results from user-supplied data mechanism: when parsing file extensions, this mechanism does not check the length of file name extensions, and copying it to a fixed-length stack-based buffer. Attacker can overwrite the return address and redirect instruction pointer to a malicious shellcode.

Analyzing the vulnerability

In this article, i use the TL-WR940N v4 firmware version 160617 to reproducing the exploitation of this vulnerability.

From main function, httpd() function is called to initializing http service. This function deploys all the endpoint of http service like ‘/’ , ‘/userRpm/LoginRpm.htm’, ‘/help/’,…. User must access the ‘/fs/’ and ‘/loginFs/’ endpoint to make the program calling httpRpmFs() function.

The Referer of request also must be equal to the host ip to pass the security check in httpDispatcher().

Analyzing httpRpmFs(), we easily show that this function is used to read the file content if this file name is in ‘/tmp/’ or ‘/web/’ folder of firmware.

Well, it look good and does not seem vulnerable. After long time analyzing this pseudocode, i find and read blogs about analyzing this vulnerability (which i add to references). They said that it is vulnerable on reading ‘passwd’ in ‘/tmp/’ folder. The firmware 211111 fixed this vulnerability so i analyzing it to proof that it is vulnerable on reading file in ‘/tmp/’, specifically ‘passwd’.

A security check in firmware 211111 prevents us reading file in ‘/tmp/’ folder if we do not have authorized access. So, i try to read ‘/tmp/passwd’.

I attach debugger to see why it crash and see that a part of request was written on stack and cause overflow.

The Return address value is overwritten by ‘3456’ which is from “cookie: authorization: 123456” of the request. A part of request overwrite return address then causing httpd crashed.

Go back with httpRpmFs(), the program will make a response contain the content of the file we want to read in ‘/tmp/’ folder after reading that file. Ensuring that the ‘Content-Type’ of response will match with file name extension of that file, sub_506790() is called to detect.

This function create a pointer which pointing to the end of file path. Then, then pointer move backward to find the ‘.’ character, and copying the part of string after ‘.’ character. The program think that all content after ‘.’ is file extension. But in this case, ‘/tmp/’ folder has file ‘passwd’, which do not have file extension. The pointer will move until it find the first ‘.’ character: the ‘.’ in its ip address in the referrer header. All character after this ‘.’ are pushed to the stack. Then, the stack is overwritten. The return address value on stack is overwritten too, so the program crash.

Reproducing the exploitation

From checksec result, the program do not have any mitigation. In some blogs, they said that the router’s ASLR is disable. So we can easily get the libc base address to build ROP gagdget. I will overwrite the return address to a ROP chain, then redirecting to a shellcode.

I borrow a shellcode from (https://www.exploit-db.com/exploits/35868) and ROPgadget from (https://www.exploit-db.com/exploits/46678) to crafting the payload of my exploit code, but after send payload, i do not receive any connection from router (which is a reverse shell). I reanalyze and find that the toupper() will uppercase all character from ‘a’ to ‘z’. My shellcode has many bytes from ‘a’ to ‘z’.

I obfuscate my shellcode with xor.encoded from pwntools, but it still has an illegal byte.

After finding on internet, i see that we can change all $t6 in shellcode to $t1, because the shellcode and rop chain do not using this register and $t1 register has the same purpose with $t6.

And finally, i craft my payload and receive a reverse shell from router.
